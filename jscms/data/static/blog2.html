Building a static version of the page on page saving

Sharing tool parametric

It wasn’t a big deal, it’s in place.

There are parameters to decide if you want to place sharing in menu bar, in the body or both, and the option to hide specific sharing channels.
The decision was to have a hide parameter (instead of show) because in this case by adding new sharing channels they’ll be automatically added and shown.

About our friendly neighbourhood web crawlers

The entire jscms backing JBLOUD was designed to be fast, effective, parametric and therefore completely javascript-driven.

I was confident that google and other search engines, used to crawl react and angular sites, would have been able to scan also this site… but apparently their code execution at crawl time is well defined, scoped, strictly limited to some code. “Trust no one”, the young Hank Moody was right 😛

On the other hand at page save time JBLOUD is saving so many static JSON files for relationships (last updated, page types and such), so what about saving also a static copy of the page?

Of course  a static copy that would be loaded (if existing) instead of the template, then always[*] updated because normal page load will load data and render everything. But the page loaded could be good for crawlers.

It was necessary to remove:

  • <body> classes
  • all <head> elements (<link>, <script> and <style>) define after jscms loader script
  • all the <nav> 
  • all the <svg> (ok, this wasn’t actually “necessary”, but useful)

… and it’s there. Achieved!

«Questa dal punto di vista seo è roba seria»

Note [*]: Not really “always updated”, actually it’s updated only unless loaded in a frame.
Updating from another domain would break SOP, in addition without updating Google translate does its work.
